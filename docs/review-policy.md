# レビュー方針

本リポジトリは個人開発アプリであり、コードの多くを生成AIが書く。生成速度に見合う品質保証を実現するため、**機械的に検証できる観点は自動化で担保し、人間のレビューは機械が苦手な高付加価値な観点（仕様の正しさ・テストの妥当性・不可逆な変更・設計判断）に集中させる**、という方針を採る。

## 機械に任せる（人間が重複してレビューしない）

以下は自動ゲートで担保する。レビューで再指摘しない：

| 観点 | 担保する仕組み |
|------|--------------|
| フォーマット・スタイル・命名 | spotless / detekt |
| シークレット漏洩 | gitleaks（CI）/ GitHub Secret Scanning + Push Protection |
| 既知脆弱性・危険API | CodeQL / detekt |
| 依存ライブラリの更新・脆弱性 | Renovate |
| リグレッション | ユニットテスト / Roborazzi / Maestro E2E |

## 人間が見る（機械で防ぎにくい）

人間のレビューは実装の行ではなく、以下の「機械化しにくい所」だけに絞る：

1. **仕様・意図の正しさ**：作るべきものを作っているか。`/feature` の仕様書レビュー停止がこのゲート。
2. **テストのオラクル**：AIが実装とテストの両方を書くと、両者が同じ前提を共有しテストが緑でも正しさを証明しない。assertの期待値が妥当かを見る（実装howではなく仕様whatを見る）。確率計算など「静かに壊れる」ロジックは特に注意。
3. **不可逆な変更**：Roomマイグレーション、既存データの変換・削除。
4. **設計・セキュリティの判断**：トークンの保存方式・ログ出力、新規依存の追加可否など。設計判断は `docs/adr/` にADRとして記録し、再指摘や判断根拠の喪失を防ぐ（雛形: `docs/adr/_template.md`）。
